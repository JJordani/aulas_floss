\textit{Patches} compartilhados entre usuários e com fabricantes\\
O importante era vender {\em computadores}\\
Custo do software era \tred{embutido} 
% Um desenvolvedor que usasse um computador poderia enviar um patch que produziu para 
% a empresa fabricante para que esta incorporasse o patch às versões futuras do software.
% Também era possível para uma pessoa enviar patches para outras. Embora não fosse
% explicitamente permitida, essa prática era aceita e até encorajada pelos fabricantes,
% pois isso melhorava a qualidade do software. 
% Essa situação não parece não diferir muito do desenvolvimento de FLOSS de hoje em 
% dia, mas há diferenças fundamentais:
% 1 - Computadores eram todos incompatíveis. O que rodava no PDP-11 não necessariamente
%     rodava no IBM/360. Embora já houvesse alguma padronização em termos de linguagens
%     cada empresa usava a sua, ou havia incompatibilidades com o SO ou a arquitetura
%     subjacentes. A construção de compiladores ainda era, até certo ponto, magia negra.
% 2 - As empresas tinham foco no hardware. O software era só aquele componente compara-
%     tivamente mais barato que era necessário para se extrair o máximo de valor a 
%     partir dos computadores. Isso passou a mudar com a redução gradual nos preços de 
%     computadores, a menor diferença de desempenho entre as máquinas disponíveis no
%     mercado e a popularização dos sistemas de compartilhamento de tempo, que ocasionou 
%     um aumento significativo na complexidade do software e, consequentemente, no seu
%     custo de produção (já que eles incluiam cada vez mais funcionalidades, apesar das
%     ferramentas de desenvolvimento cada vez mais evoluídas). 
% 3 - Não havia a facilidade de se compartilhar código que passou a existir com o 
%     advento da Internet. Nessa época, compartilhar código era muito mais difícil, exi-
%     gindo o envio de disquetes ou de fitas magnéticas. Qualquer um desses era caro
%     de enviar, além de pouco conveniente.

% Com a popularização e padronização das linguagens de alto nível, passou a ser possível 
% rodar um mesmo programa em máquinas de fabricantes distintos e tornou-se mais fácil
% construir programas mais complexos. Neste cenário, três fatores levaram os fabricantes
% de hardware a se preocupar com software: (i) a emergente indústria de software; 
% (ii) leasing de máquinas, que funcionava como um aluguel e não trazia nenhum lucro
% em termos de software; e (iii) a capacidade de vários clientes dos fabricantes de
% hardware de atender suas próprias demandas, em termos de software. Software cada vez 
% mais tornava-se um diferencial competitivo e simplesmente deixar qualquer um 
% beneficiar-se disso parecia uma má estratégia de negócios. Competidores poderiam tomar
% como base o software produzido por uma empresa, melhorá-lo e vendê-lo como seu próprio
% produto. Fabricantes passaram então a dificultar a obtenção do código fonte dos programas
% e a exigir que a lei de copyright se fizesse cumprir. Na segunda metade da
% década de 70, popularizou-se a prática de vender licenças de uso, como as que 
% são usadas até hoje pela Microsoft.

% Na guerra dos browsers, o Netscape Navigator perdeu mercado rapidamente
% devido à política da Microsoft de distribuir o Windows junto com o Internet
% Explorer. Adicionalmente, este último foi integrado ao Windows de tal
% maneira que tornou-se impossível simplesmente livrar-se dele. Apesar de
% vários processos na justiça, a manobra da Microsoft foi eficaz. Isso
% preocupou os executivos da Netscape não pelas razões óbvias. A empresa
% não obtinha lucros a partir da distribuição do browser, já que esta era
% feita gratuitamente. Seus lucros eram provenientes de software para
% servidores. Era importante combater a dominação da Microsoft principalmente
% porque isso poderia implicar em um monopólio não apenas no software
% que rodava nos clientes. Esta situação coincidiu com a popularização 
% do Linux, que já contava com algumas distribuições mais amigáveis, como
% o Red Hat Linux, e com o texto que apresentou a forma ``open source'' de
% desenvolver código para o mundo, ``The Cathedral and the Bazaar'' de Eric
% S. Raymond. Esse texto admitidamente foi o pontapé que a Netscape precisava
% para liberar o código fonte do seu browser, em janeiro de 1998, e formar o 
% que se tornaria a Mozilla Foundation.
